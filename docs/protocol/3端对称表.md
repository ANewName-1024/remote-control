## RTT/Skew 指标层 (`/api/agents`)

| 字段 | 计算 | 含义 | 修复链路 |
|------|------|------|----------|
| `lastRttMs` / `rttAvgMs` / `rttMaxMs` | `Date.now()/1000 - msg.ts` × 1000 | **Legacy**: 单向 RTT，**含** wall clock skew | 跟 agent `rtt_avg` 比对, 差大 = skew |
| `lastClockSkewMs` / `skewAvgMs` / `skewMaxMs` | `Date.now() - msg.ts * 1000` | **New**: wall clock skew term (含 1-way RTT 约 16-50ms) | >100ms 持续 = NTP broken. Linux: `apt install chrony && systemctl enable --now chrony`. Windows: `sc config w32time start= auto && w32tm /config /manualpeerlist:"ntp.aliyun.com pool.ntp.org" /syncfromflags:manual && reg add HKLM\..\W32Time\Config /v PhaseCorrect /d 1 /f && reg add HKLM\..\W32Time\Config /v MaxAllowedPhaseOffset /d 1 /f && net stop w32time && net start w32time && w32tm /resync /force` |
| `lastServerProcMs` / `procAvgMs` / `procMaxMs` | `process.hrtime.bigint()` delta | **New**: server 处理时延 (monotonic, 跟 wall clock 无关). 健康 < 5ms | 涨 = Node event loop 阻塞 (CPU 100% / GC / sync IO) |

## 真实问题诊断 — 2026-06-10 18:30

| 现象 | 数字 | 原因 |
|------|------|------|
| `w32tm /stripchart o:` (旧) | `+0.6s` (5+ servers 一致) | Windows 端 W32Time 服务**从未启动**, 系统时钟跟 UTC 漂移 |
| `rttAvgMs` (旧) | `680ms` | 算式 `(Date.now()/1000 - msg.ts) * 1000` = clock skew + 1-way RTT |
| Agent `rtt_avg` (旧) | `22ms` | Agent 端**真** round-trip RTT (用自己时钟) |
| 真实网络 RTT (ICMP) | `16ms` | 本机 → VPS 物理延迟 |
| 真实 clock skew (新 `clockSkewMs`) | `~600ms` | 跟 NTP 漂移对得上 |
| `lastServerProcMs` | `0.14-0.25ms` | server 处理时延极低, 不是瓶颈 |

**修复后**:
- `w32tm /stripchart o:` = `-00.0069s` (~7ms 抖动, 健康)
- `rttAvgMs` = `27ms` (跟 ICMP 16ms + 一些 buffer 对得上)
- `clockSkewMs` = `27ms` (基本是 1-way RTT, 无 skew 残留)
